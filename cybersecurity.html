Basic Home Network Enumeration

Remediation Notes

Remediation Notes

To reduce device discovery and limit exposed information during network scans, such as those performed with Angry IP Scanner, consider the following security practices:

Enable Network Segmentation (VLANs)

Separate devices into different network segments so they cannot easily discover or communicate with each other.

Example command to view current network configuration on Linux:

ip addr

This shows your assigned IP addresses and interfaces.

Use Wi-Fi Client Isolation

Enable client isolation or guest network isolation on your router. This prevents devices on the same wireless network from communicating directly with each other.

To identify the gateway (router) address:

ip route

The gateway IP is usually something like 192.168.1.1.

Configure Host-Based Firewalls

Use system firewalls to block unnecessary services and limit responses to scans.

Example using UFW on Linux:

sudo ufw enable
sudo ufw deny 23
sudo ufw allow 22

This blocks Telnet and allows SSH.

Disable ICMP Echo Responses (Ping)

Blocking ping responses can make systems less visible to basic scanners.

Example on Linux:

sudo sysctl -w net.ipv4.icmp_echo_ignore_all=1

Limit Exposed Services

Regularly check which ports are open on your system.

Example command:

ss -tuln

This lists listening ports and services.

Check Active Devices on Your Network

Understanding what devices exist on your network helps identify unknown hosts.

Example command:

arp -a

This shows devices that have communicated with your machine on the local network.

Learning Tip

Students can also practice enumeration themselves:

nmap -sn 192.168.1.0/24

This performs a simple network discovery scan similar to what Angry IP Scanner does.

Wireshark Investigation

FTP Brute Force Attack

Step-by-step packet analysis showing how weak credentials and plaintext FTP led to a web-shell upload.

Case Notes

Source: Wireshark packet capture review

1. Initial Network Traffic Observation

The first step was to inspect FTP communication between the two hosts in the packet capture.

Source IP:      192.168.0.147
Destination IP: 192.168.0.115

The trace showed a normal TCP handshake followed by an FTP banner response:

220 Hello FTP World!

This confirmed the FTP service was active and accepting connections.

2. Username Enumeration

Shortly after the banner response, repeated login attempts targeted the same FTP account:

USER jenny
USER jenny
USER jenny

The repeated requests show the attacker already knew, or had enumerated, the username jenny.

3. Password Brute Force Attempts

The server repeatedly answered with:

331 Please specify the password.

That was followed by a sequence of common dictionary passwords:

PASS 12345
PASS password
PASS 12345678
PASS dragon
PASS sunshine
PASS qwerty
PASS football
PASS computer
PASS monkey
PASS michael
PASS shadow
PASS letmein
PASS daniel

This pattern is consistent with automated brute-force tools such as Hydra, Medusa, or Ncrack.

4. Successful Login

Eventually the correct password appeared in the capture:

PASS password123
230 Login successful.

This confirmed the compromised credentials:

Username: jenny
Password: password123

5. FTP Post-Login Activity

Once authenticated, the attacker began working inside the Apache web root:

257 "/var/www/html" is the current directory

This directory is critical because files placed there can be served directly by the web server.

6. File Upload and Permission Changes

The capture then showed an uploaded PHP file:

STOR shell.php

Wireshark also revealed the FTP data transfer and a permission change:

FTP-DATA: STOR shell.php
SITE CHMOD 777 shell.php

That sequence strongly suggests a web shell deployment. Permission 777 grants read, write, and execute access to everyone.

7. Attack Chain Summary

The packet capture reconstructs the attack path clearly:

1. FTP service discovered
2. Username jenny targeted
3. Dictionary password attack launched
4. Login succeeds with password123
5. Attacker moves into /var/www/html
6. shell.php is uploaded
7. chmod 777 shell.php is applied
8. The shell can be triggered through the web server

8. Security Weaknesses Identified

The compromise succeeded because several weak controls aligned:

Weak password: password123 was easily guessed from a dictionary list.

Plaintext FTP: credentials and commands were visible directly in the packet capture.

Web root write access: the FTP user could upload straight into /var/www/html, enabling a web-shell drop.

9. Defensive Recommendations

Several controls would reduce or prevent this attack path:

- Enforce strong password policies
- Replace FTP with SFTP or FTPS
- Restrict upload access to non-web-root directories
- Alert on repeated failed logins
- Use a WAF to inspect and block suspicious uploads

10. Real Pentester Skill

PCAP files are valuable because they can expose credentials, malware downloads, command execution, attacker tooling, and lateral movement. In unencrypted traffic, one capture can reveal the full compromise timeline.
